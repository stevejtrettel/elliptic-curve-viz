# Survey: `lifting-modp`

*Exploration report generated 2026-07-04 as input to the elliptic-curve-art rewrite. The original (functional but messy) renderer.*

## 1. Directory structure & tech stack

**Stack**
- **Language:** plain ES-module JavaScript (no TypeScript, no JSX). No type checking.
- **Build/dev:** Vite 5 (`package.json` scripts: `dev`/`build`/`preview`). `index.html` at repo root loads exactly **one** scene module via `<script type="module">` — you edit that one line to pick which scene renders. Currently points to `./js/scenes/cplxMult/latticeHex.js`.
- **Libraries:** `three@0.172.0`, `three-gpu-pathtracer@0.0.23`, `mathjs@14.4.0` (mathjs appears to be an unused/legacy dependency — not imported in the core render path). Uses three's bundled `examples/jsm` for `OrbitControls`, `lil-gui`.
- **Python side** (`python/pycode/`, ~3400 lines): SageMath/number-theory code (`ecqf.py`, `ecfp.py`, `qfs.py`, `ringclasses.py`, `nt.py`, `modularpolynomials.py`) that computes the elliptic-curve-over-finite-field point sets and CM/lattice data. This is the *upstream data generator*; it is not called at render time. A `python/userguide.ipynb` exists.

**Top-level layout**

## 2. Data input format

Point sets live in `data/<disc>/<n>.js` (e.g. `data/-8/2.js`). Each is an ES module with a single `export default` of an **array of `[x, y]` pairs of floats**:

```js
export default [[0.61111111,0.39283710],[0.11111111,0.15713484], ... ];
```

**Coordinate meaning (exact):** the pairs are **fractional/normalized coordinates in the fundamental domain of the CM lattice**, in the basis `{1, τ}`. I.e. a stored point `[x,y]` corresponds to the complex/planar point `x·(1,0) + y·(0,1)` where the lattice is generated by `1` and `τ`. They are *not* pixel coordinates and *not* yet on the torus. The x-values cluster on rationals with denominator = the field size (e.g. `data/-8/2.js` shows denominators of 18 → this is the group of points), and y-values are multiples tied to `τ.y`.

Each discriminant folder carries a **`tau.js`** module defining the lattice + profile curve (see §3). Newer/rect-vs-para variants: `data/-20/rect/` and `data/-20/para/` each hold `data.js`, `data-sm.js` (a coarser subsample used for lighter renders), and `tau.js`.

Scenes import both:
```js
import {coordCurve, latticeData} from "/data/-8/tau";
import data from "/data/-8/2";
```
(Absolute `/data/...` and `/js/...` import paths resolved by Vite from project root.)

Note the mismatch to watch in a rewrite: point files are *pure data*, but the **lattice geometry AND the S² profile curve are hand-authored per discriminant inside `tau.js`** — they are code, not data.

## 3. Core mathematics pipeline

Flow: **2D lattice coords `[x,y]`** → (`fromTauCoords`) **rectified plane coords `(u,v)`** → (`isometricImage`) **point in S³** → (`stereoProj`) **point in R³**.

### 3a. Lattice / modulus and the "rectifying" change of coordinates — `data/<disc>/tau.js`

Each `tau.js` exports:
- `tau: Vector2` — the lattice modulus τ (e.g. `-8`: `(0,√2)`; `-3`: `(-1/2,√3/2)` hexagonal; `-20 rect`: `(0,√5)`).
- `curveArea`, `curveLength` — the **two side lengths of the flat torus's rectangular fundamental domain** in the Hopf/Pinkall parametrization. Stored as `area` (= A) and `length` (= L). For `-8`: `A = 4π/3`, `L = 4π√2/3`. The Pinkall lattice is generated by `(2π, 0)` and `(A/2, L/2)`.
- `coordCurve(t) → {phi, theta}` — the **profile curve on S²** (spherical coords), 0≤t≤2π. For rectangular tori it's a *circle of latitude*: constant `phi = π/2 ± a`, `theta = t`, where `a = acos(2√2/3)` for `-8` (length = 2πR of the latitude circle ⇒ R = 2√2/3 ⇒ tilt angle a). For the **hex torus (`-3`)** the profile curve is a genuine *wavy* Pinkall curve: `phi = π/2 + a·b·cos(n·t)`, `theta = t + a·sin(2n·t)` with hand-tuned `(a,b,n)` — this is what makes a non-round (non-Clifford) flat torus that closes up hexagonally.
- `fromTauCoords(data) → Vector2` — maps a stored `[x,y]` (in the `{1,τ}` fundamental domain) into the **Pinkall lattice coordinates** `(u,v)`. This is *not just a rescale* (comment stresses this): it applies an `SL₂(ℤ)` change of representation + a rotation + rescale, then `×2π`.
  - `-8` version (`data/-8/tau.js:36`): rotate by `θ = arctan√2` (`sin=√(2/3), cos=√(1/3)`), divide by `√3`, `×2π`.
  - `-3` version (`data/-3/tau.js:42`): since `Re(τ)=1/2`, it degenerates to a pure `×2π` rescale.
  - `-20 rect` (`data/-20/rect/tau.js`): rotate by `arctan√5`, divide by `√6`, `×2π`.

### 3b. Pinkall/Hopf torus construction + conformal parametrization — `js/items/HopfTorus.js`

`class HopfTorus(coordCurve, latticeData)`. The heart is building an **isometry** from the flat rectangular domain to the Pinkall surface. Three numerically-integrated helpers (all Riemann sums, fixed step):

- **`arcLength(t)`** (`HopfTorus.js:54`) — arclength of the profile curve on S² up to t, using metric `ds² = sin²φ·dθ² + dφ²`.
- **`inverseArc(L)`** (`:78`) — inverts arclength: finds t whose profile-curve arclength from 0 equals L (marches with `dt=2π/3000` until cumulative length exceeds L).
- **`fudgeFactor(t)`** (`:36`) — the Pinkall "twist" correction: `∫₀ᵗ sin²(φ/2) dθ` (step `dt=0.00025`). This is the horizontal-lift holonomy term that keeps the parametrization conformal/closing-up correct.

**`isometricImage(pt)`** (`:129`) — the key map, plane `(u,v)` → R³:
```
pt = toFundamentalDomain(pt)      // wrap v into the strip (see pain points)
s = pt.x ; v = pt.y
t = inverseArc(2*v)               // vertical coord → param along profile curve by arclength
{phi,theta} = coordCurve(t)
f = fudgeFactor(t)
P = toroidalCoords(theta + s - f, s - f, phi/2)   // lift into S³ (Vector4)
return stereoProj(P)              // → R³
```
So: **u = fiber (Hopf circle) direction**, **v = base-curve (profile) direction**, and the `-f` shear makes it a true isometry from a *rectangle*.

**`toroidalCoords(a,b,c)`** (`utils.js:4`) — the Hopf/Pinkall embedding S¹×S¹→S³⊂R⁴:
```
x=cos(a)sin(c), y=sin(a)sin(c), z=cos(b)cos(c), w=sin(b)cos(c)
return Vector4(x, z, -y, w)      // axis reorder
```
Here `c=φ/2` sets the "latitude" (torus tube radius in S³), `a=θ+s−f` and `b=s−f` are the two circle angles. This is exactly the standard Hopf-fibration torus: fixing the base point (φ,θ) and sweeping `s` traces one Hopf fiber (a great circle in S³).

**`stereoProj(pt)`** (`utils.js:19`) — S³→R³ from the north pole in the z-slot:
```
return Vector3(pt.y, -pt.x, pt.w) / (1 - pt.z)
```
(An "orig" version projecting `(x,y,w)/(1−z)` is commented out just above — projection axis is a hand-tweaked knob.)

**`sphCoords({phi,theta})`** (`utils.js:26`) — standard S² embedding, used only for drawing the base sphere/curve in `HopfPreimage`.

`HopfTorusNonIso.js` is the same pipeline **minus** `inverseArc` and `fudgeFactor`: `surface(s,t)=stereoProj(toroidalCoords(θ+s, s, φ/2))` directly (`HopfTorusNonIso.js:36`). It exists to visually contrast the naive non-conformal parametrization.

### 3c. Surface tessellation

`getSurface()` (`HopfTorus.js:149`) builds a `ParametricGeometry` over `(s,t)∈[0,1]²` scaled to `[0,2π]²`, at `res=256` (so 256×256 grid). `getSubSurface()` uses 512×512. Normals via finite differences (three's ParametricGeometry, `EPS=1e-5`).

## 4. Rendering

- **Points of the elliptic curve:** each is an **individual `THREE.SphereGeometry` `Mesh`** placed at `isometricImage(fromTauCoords([x,y]))`. See `HopfTorus.getPoint()` (`HopfTorus.js:246`) and the scene loop in `lifts/-8/2/torus.js:51-54`. **No instancing** — one Mesh + one material per point (Box and Dodecahedron variants are commented out). Radius is scaled by `1 + q.lengthSq()` to compensate for stereographic distortion (points far from projection center get drawn bigger so they *look* uniform in S³).
- **Torus surface:** rendered as a solid `Mesh` from `getSurface()`, `MeshPhysicalMaterial`, optionally `glass` (transmission 0.99, ior 1.05). Default color `colors.glass = 0xc9eaff`.
- **Curves/fibers/gridlines/lifts:** tubes via the custom **`VarTubeGeometry`** (variable radius, radius also `×(1+lengthSq)` distortion-corrected), Frenet-frame swept, `radialSegments=16`. See `getLift`/`getFiberAt`/`getEdgeAt`/`getGridlines` (`HopfTorus.js:184-244`).
- **Fundamental-domain (flat) scenes** (`fd.js`): use `items/FD.js` — a sheared `BoxGeometry` parallelogram plus `SphereGeometry` data points laid flat in R³.
- **Materials:** centralized in `makeMaterial(color, glass)` (`utils.js:40`): `MeshPhysicalMaterial`, `roughness 0.5, metalness 0`; glass branch adds transmission. Palette `colors`/`redShades`/etc. in `utils.js:60-101`.
- **Path tracing:** **every** scene integrates `three-gpu-pathtracer`'s `WebGLPathTracer` (151 of 160 scene files). Standard setup: `renderScale = max(1/dpr, 0.5)`, `tiles=3`, `bounces=30`, `ACESFilmicToneMapping`, `GradientEquirectTexture` environment, a `PhysicalSpotLight`, a white `BoxGeometry` ground plane. `OrbitControls` with `controls 'change' → pathTracer.updateCamera()`. A GUI with a single "saveit" PNG-download button. `animate()` just calls `pathTracer.renderSample()` each frame (progressive accumulation).

## 5. Parameters / knobs

- **Lattice/modulus:** `tau` and the derived rectangular domain sides `area (A)`, `length (L)` — per `tau.js`.
- **Profile curve shape:** `coordCurve` `phi/theta` — for hex tori the `(a,b,n)` amplitude/frequency triple (several commented alternatives in `data/-3/tau.js:12-34`); for rectangular tori the latitude offset `a` and a `±a` sign choice ("skinnier torus in projection").
- **Coordinate rectification:** rotation angle + rescale divisor inside `fromTauCoords`.
- **Stereographic projection axis/center:** which two coords go to R³ and the `(1−pt.z)` pole — two variants in `utils.js` (one commented out); effectively the projection center is a knob.
- **Point size / tube radius:** `radius` args (typ. `0.045`, `0.025`) with distortion factor `1+lengthSq`.
- **Rotation in S³:** encoded implicitly via the `θ+s−f`, `s−f` angle offsets and the `toroidalCoords` axis reorder `(x,z,-y,w)`; there is **no explicit SO(4) rotation parameter** — reorientation is done ad hoc.
- **Tessellation resolution:** `res=256` (surface), 512 (subsurface), tube `2·res × 16`.
- **Render:** camera position/target, spotlight intensity, `bounces`, `renderScale`, `tiles` — all hardcoded per scene.

## 6. Pain points (for the rewrite)

1. **Massive scene duplication.** ~160 near-identical scene files, each a full standalone app that re-inlines the entire renderer + pathtracer + spotlight + ground + GUI + resize/save boilerplate (~200 lines of copy-paste per file). Only the top ~50 lines (what objects get added) differ. No shared "app/harness" module. `PathTracerControls.js`/`loadingScreen.js` look like an abandoned attempt at factoring this out.
2. **Per-discriminant math lives in data as hand-authored code.** `fromTauCoords`, `coordCurve`, and `A/L` are separately hand-derived and hardcoded in each `tau.js`, with the SL₂(ℤ) reasoning only in comments. Rederiving τ → (A, L, rotation, profile latitude) from first principles should be a single reusable function; right now `-8`, `-20`, `-3` each reimplement it slightly differently (e.g. `-8` hardcodes `√(2/3)` while `-20` computes `sin(atan(√5))`).
3. **Numerical integration is brute-force and re-run constantly.** `isometricImage` calls `inverseArc` (marches up to 3000 steps) **and** `fudgeFactor` (up to `t/0.00025` ≈ tens of thousands of steps) **on every single point and every surface vertex** (256² or 512² vertices). No precomputed arclength table / no caching / no shared lookup. This is O(N·steps) and is the obvious performance bottleneck; a precomputed monotone arclength LUT with interpolation would replace both.
4. **`toFundamentalDomain` is admittedly broken/partial.** Comment `//FIX THIS FUNCTION TO BE BETTER, LATER` (`HopfTorus.js:106`); only wraps the `v` (vertical) direction with a `while` loop, relies on all input points being positive, and has large commented-out blocks for the horizontal wrap. Fragile for any point outside the assumed strip.
5. **No instancing for points.** Every curve point is a separate Mesh+Material clone — wasteful for large point sets (field sizes can be big); should be `InstancedMesh` with per-instance scale.
6. **Axis-swizzling everywhere.** Coordinate reorders like `(x,z,-y,w)` in `toroidalCoords`, `(pt.y,-pt.x,pt.w)` in `stereoProj`, `(p.x,p.z,-p.y)` in `HopfPreimage` — magic axis permutations scattered across files to make things "look right," with alternative versions commented out. No single documented orientation convention.
7. **Dead/duplicated code.** `scenes/OLD/`, `python/older/`, unused `mathjs` dependency, `HopfTorusNonIso` duplicating `HopfTorus`, commented-out `PhysicalCamera`/area-light blocks pasted into most scenes.
8. **Global module side effects.** Each scene mutates the DOM/`document.body` and starts its own RAF loop at import time; can't compose or unit-test. A rewrite wants a `HopfTorus` pure-math core (no three.js) + a thin renderer layer + a data-driven scene registry.

Key files to port first: `js/items/utils.js` (coord transforms), `js/items/HopfTorus.js` (lines 36-145 = all the real math), and the `fromTauCoords`/`coordCurve` logic from `data/*/tau.js` (generalize into one lattice→Pinkall function).
